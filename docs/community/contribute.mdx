请仔细阅读以下准则。

## 贡献方式

贡献形式多样，不仅限于编写代码。
如有疑问请参阅[§ 支持][support]。
若可提供资金支持，请查看[§ 赞助][sponsor]。

### 改进文档

作为用户，您最适合帮助我们完善文档。
包括修正拼写错误、修复表述问题、优化说明文字、新增示例等。
所有MDX文档均位于`docs/`目录。

可在本地运行文档站点，详见下文[¶ 网站][site]。

### 完善议题

部分议题可能信息不全、无法复现或存在谬误。
您可以通过补充信息使其更易解决。
现有议题可能正需要您独特的经验或见解。

### 编写代码

我们非常欢迎代码贡献。
建议在提交拉取请求前，先通过提问或创建议题来报告错误或建议新功能。

## 提交议题

* 议题追踪器仅用于问题讨论
  技术支持请使用讨论区
* 提交新议题前请先搜索议题追踪器（包括已关闭议题）
* 确保使用最新版本的相关包
* 使用清晰明确的标题
* 包含尽可能多的信息：问题重现步骤、错误信息、版本号、操作系统等
* 您在议题中投入的时间越多，我们越能有效帮助您
* 最佳的议题报告是能证明问题的失败测试用例

## 提交拉取请求

* 项目结构信息详见下文[¶ 项目][project]，包括测试方法和站点构建说明
* 重大变更建议先在议题中讨论，避免无效劳动
* 对于复杂任务，应尽早让社区看到工作成果以获取反馈
* 新功能需附带测试和文档
* 请勿包含无关变更
* 提交代码前请运行`npm test`进行测试
* 在描述中充分说明为何应合并您的拉取请求：说服我们是您的责任

## 项目结构

### 目录架构

MDX采用monorepo结构。
所有包位于`packages/`目录。
文档存放在`docs/`目录。

### 测试流程

运行测试前请先执行`npm install`，然后执行`npm test`。
这将检查代码风格、单元测试和类型定义等所有环节。

### 网站构建

构建网站前请先执行`npm install`，然后执行`npm run docs`。
生成的网站内容位于`public/`目录。

### 版本发布

发布新版本需执行以下步骤：

1. 更新各包的`version`号（注意同步更新monorepo内部依赖范围）：
   ```sh
   npm version minor --workspaces --no-git-tag-version
   ```
2. 提交代码并使用版本号（不带v前缀）作为提交信息：
   ```sh
   git commit --all --message 1.2.3 && git tag 1.2.3 && git push && git push --tags
   ```
3. 发布到npm仓库：
   ```sh
   npm publish --workspaces
   ```
4. 在GitHub上添加版本更新日志：
   ```sh
   open https://github.com/mdx-js/mdx/releases
   ```
